# References — Mobile / Screens

> Topic reference folder. Agents read this folder **only when the active task needs it** (`../../../system/CONTEXT_MANAGEMENT_RULES.md`, `TOKEN_OPTIMIZATION_RULES.md`) — never the whole tree. Starts empty; material is added as the project develops. **Do not create fake reference content.**

## What belongs here

Screen-level reference material for the mobile app: example screen implementations, layout patterns, screen-specific interaction notes, and mockup links for particular screens.

## Supported file types

Markdown (`.md`) for patterns/notes; small code snippets in fenced blocks; images (`.png`/`.svg`) for screen mockups (linked where large).

## Naming conventions

`kebab-case` by screen (`profile-screen.md`, `onboarding-flow.md`). Keep one screen/pattern per file.

## How agents should use it

Read when building or modifying that screen, alongside `../../../skills/mobile/mobile-design-system` and `mobile-navigation`. Use as a pattern to follow, not a spec to copy blindly.

## References are not requirements

Material here is **context, not commitment.** A screen reference does **not** automatically become a requirement or approved design. Requirements/tasks and gate approval govern what is built (`../../../system/QUALITY_GATES.md`). Cite a reference when it informs the screen; never let it silently expand scope.
