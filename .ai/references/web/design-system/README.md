# References — Web / Design System

> Topic reference folder. Agents read this folder **only when the active task needs it** (`../../../system/CONTEXT_MANAGEMENT_RULES.md`, `TOKEN_OPTIMIZATION_RULES.md`) — never the whole tree. Starts empty; material is added as the project develops. **Do not create fake reference content.**

## What belongs here

Web design-system reference material: tokens, scales, reusable component/primitive patterns, theming (light/dark), and typography conventions.

## Supported file types

Markdown (`.md`) for patterns/notes; small code snippets in fenced blocks; images (`.png`/`.svg`) for component references (linked where large).

## Naming conventions

`kebab-case` by concern (`tokens.md`, `theming.md`, `component-primitives.md`).

## How agents should use it

Read when building UI components/theme, alongside `../../../skills/web/web-design-system` and `web-theme`. Follow the token/primitive patterns; don't hand-roll ad-hoc styling.

## References are not requirements

Material here is **context, not commitment.** A design-system reference does **not** automatically become a requirement or approved design. Requirements/tasks and gate approval govern the build (`../../../system/QUALITY_GATES.md`). Cite when it informs the work; never let it silently drive scope.
