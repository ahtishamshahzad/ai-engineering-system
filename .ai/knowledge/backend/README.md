# Knowledge — Backend

> Durable, reusable backend engineering knowledge. Read only when the task needs it. **Not a tutorial dump.** Every entry carries a trust label; version-sensitive entries carry the metadata block — see `../README.md`. Starts empty.

## What belongs here

- Reusable API/auth/jobs/integration patterns and their trade-offs.
- Express/NestJS framework constraints and version-specific pitfalls (with versions).
- Third-party integration lessons with **future value** (quirks, rate limits, idempotency needs).

## What does NOT belong here

- Generic framework tutorials — link the docs.
- Project-specific endpoint/contract notes → `../../references/backend/`.
- Canonical rules → `../../system/`. **No secrets or credentials — ever.**

## Quality & source requirements

- Reusable, non-obvious; cite the source (docs, provider changelog, incident).
- Record `affected_versions` for framework/library notes; undated version claims are **unverified**.
- Label trust: verified-current / project-convention / opinion / deprecated / unverified.

## Update & deprecation

- Re-verify against current framework/provider versions; mark superseded patterns **deprecated** with a successor.

## How skills reference this

- `../../skills/backend/*` link here for version-sensitive/integration knowledge, keeping the skills durable and version-free.
