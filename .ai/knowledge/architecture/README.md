# Knowledge — Architecture

> Durable, reusable architecture knowledge. Read only when the task needs it. **Not a tutorial dump.** Every entry carries a trust label and, if change-prone, the metadata block — see `../README.md`. Starts empty; populated as durable lessons accrue.

## What belongs here

- System-design principles and boundary/decomposition patterns proven useful.
- ADR-style architecture decisions worth **reusing across projects** (project-specific ADRs stay in `../../projects/current/`).
- Integration-architecture patterns (how services/contracts compose) with their trade-offs.
- Non-obvious structural constraints and the reasoning behind them.

## What does NOT belong here

- Generic "what is microservices" tutorials — link the source instead.
- A single project's architecture doc → `../../projects/current/` / `../../templates/ARCHITECTURE.md`.
- Canonical rules → `../../system/`.

## Quality & source requirements

- Reusable, non-obvious, and stated with its limits and the *why*.
- Cite the source (ADR, spec, postmortem). Uncited technical claims are **unverified notes**.
- Label trust: verified-current / project-convention / opinion / deprecated / unverified.

## Update & deprecation

- Update the entry and its `verified` date when the picture changes; mark superseded patterns **deprecated** with a successor link (don't silently delete).

## How skills reference this

- `../../skills/architecture-design` and `repository-architecture` stay instructional and **link** here for reusable pattern knowledge rather than embedding it.
